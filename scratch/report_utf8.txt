To understand why this algorithm is highly effective for tabular HR data, one must understand the mathematics of an individual decision node. An individual Decision Tree operates by splitting the dataset into smaller subsets geographically (e.g., separating employees who take 'Public Transport' from those who take 'Staff Buses'). The algorithm evaluates the optimal splitting point by calculating the Gini Impurity at each potential node. The Gini Impurity (G) measures the probability that a randomly chosen element would be incorrectly classified if it were randomly labeled according to the distribution of labels in the node:Where C is the number of classes (in this binary case, 2: 'Active' vs 'Resigned'), and pi is the probability of picking a datapoint with class i. The algorithm seeks to maximize Information Gain the reduction in Gini Impurity from the parent node to the localized child nodes.However, individual Decision Trees are notorious for "overfitting" they mathematically memorize the exact training data structure, leading to very low bias but extremely high variance when introduced to unseen future employee data. The Random Forest explicitly solves this Bias-Variance tradeoff via Bootstrap Aggregating (Bagging). Instead of building one massive, complex tree, the algorithm builds hundreds of "shallow" trees (controlled via the `max_depth` hyperparameter). Each tree is trained on a random, bootstrapped subset of the 5,000 employee records, and only a random subset of features is considered for splitting at each node. During deployment, when a new employee CSV file is uploaded, their data is passed through all 500 trees. Each tree casts a "vote" on whether the employee will resign, and the majority consensus dictates the final classification. This ensemble approach dramatically stabilizes the variance, creating a robust, generalized predictive boundary.3.6.2 Algorithmic Resolution for Class ImbalanceA critical theoretical challenge in HR Analytics is the natural occurrence of Class Imbalance. In a healthy garment factory, the vast majority of employees (e.g., 85%) will stay, while a minority (e.g. 15%) will resign. If a machine learning algorithm is fed this raw distribution, the underlying loss-function mechanics will mathematically bias toward the majority class. The algorithm quickly learns that simply predicting "Active" for every single employee guarantees an 85% generalized accuracy rate.To combat this, rather than generating unverified synthetic data which strays from real organizational records, this project utilized algorithmic Class Weighting inside the Random Forest hyperparameters. By passing the argument `class_weight='balanced_subsample'`, the predictive engine alters the mathematical penalty of its loss function during training. Errors made against the minority class (predicting a 'Resigned' employee as 'Active'ùa False Negative) are penalized exponentially higher than errors made against the majority class. This forces the decision boundaries inside the Gini Impurity splits to aggressively encapsulate the behavioral footprints of the rare 'leavers', ensuring the system remains highly sensitive to resignation risks regardless of the overwhelming volume of active staff.3.6.3 Model Comparison and Selection JustificationPrior to committing to the Random Forest architecture, a rigorous comparative analysis was conducted across four distinct classification algorithms: Logistic Regression, standard Decision Trees, Random Forest, and Gradient Boosting.The initial baseline evaluation produced the following Recall (Sensitivity) scores for correctly identifying the critical Class 1 (At-Risk Leavers):Decision Tree: 80.53%Logistic Regression: 89.86%Random Forest (Baseline): 88.44%Gradient Boosting: 91.68%At first glance, Gradient Boosting technically achieved the highest initial baseline recall array. However, Gradient Boosting builds trees sequentially, where each tree attempts to correct the residual errors of the previous one. In highly noisy, categorical tabular datasets (like localized demographic HR records), this sequential error-correction makes Gradient Boosting highly unstable and prone to extreme overfitting if not perfectly monitored. Random Forest, acting in parallel, is vastly more robust to noise and computationally lighter to deploy inside an offline Streamlit web container.To bridge the 3% baseline gap, the Random Forest model was subjected to an aggressive RandomizedSearchCV Hyperparameter Tuning protocol using 5-Fold Stratified Cross-Validation. The grid search specifically minimized pure Accuracy in favor of strictly optimizing for the `Recall` scoring metric. The best parameters extracted were: `{'n_estimators': 500, 'max_depth': 10, 'class_weight': 'balanced_subsample'}`.When this mathematically tuned Random Forest was exposed to the 20% unseen, hold-out testing data, it achieved a phenomenal 96.15% Recall Rate. This overwhelming performance increase decisively justified selecting Random Forest. It provided the exact strategic capability HR required: an algorithm capable of actively catching 96% of all impending resignations, utilizing an interpretable, parallel-processing architecture perfectly suited for lightweight local deployment.3.7 Testing StrategyTo guarantee correct integration of the machine learning model with the web dashboard, and to verify the accuracy of the prediction outputs, a comprehensive test plan was formulated.3.7.1 Test Environment & ToolsTo ensure the system remains stable, the following environment and tools were utilized:Hardware: Intel Core i5 (minimum) with 8GB RAM to handle Pandas dataframes without lag, and at least 500MB of storage.Software: Windows 10/11, Python 3.9+, and Chrome/Edge for cross-browser UI testing.Testing Methods: Manual UI testing was conducted to verify button flow. After building and testing the system, the project definitely yielded some interesting findings that mix data science directly with Human Resources.First off, I proved that Random Forest Ensembles are pretty much the optimal architecture for this kind of structured, tabular HR data. A lot of current research focuses on Neural Networks, but those are computationally heavy and act like "black boxes" that you can't easily explain to a manager. By using a Random Forest classifier instead, my system mapped out complex relationshipsùlike how 'Transport Mode' interacts with 'Service Years'ùwithout slowing down the computer during deployment. Secondly, my project tackled what we call the "Accuracy Paradox" in HR modelling. In a normal garment factory environment, the vast majority of employees (usually around 85-90%) will stay. This creates a very lopsided dataset. If I had tuned my model just to achieve high "Accuracy," it would easily hit a 90% success rate just by guessing that every single employee stays forever. Of course, that completely defeats the purpose because it misses the exact 10% of people who are actually going to quit. To fix this paradox, I evaluated the system using a Confusion Matrix:True Positives (TP): The model correctly flags a person who is about to leave. (This is the ultimate goal for HR).True Negatives (TN): The model correctly predicts that a loyal employee will stay. (Good, but standard).False Positives (FP): The model gets confused and flags a loyal employee as a flight risk. (A minor annoyance, a manager might waste 5 minutes checking in on a happy employee).False Negatives (FN): The model completely misses an at-risk employee. (This is a catastrophic failure because the employee walks out and production stops).My testing proved that algorithmically tuning the Random Forest specifically for Recall (Sensitivity) mathematically forces the model to minimize False Negatives. Essentially, I decided to trade a small number of False Positives so that I could ensure a massive capture rate of True Positives. This proved to be the most practical approach for real HR interventions.Finally, the project showed that Geographic and Mobility Proxies are massive predictors for turnover in Sri Lanka. I didn't have access to financial data like exact salaries because of company security policies. Despite that, metrics like the employee's 'District' combined with whether they used a 'Staff Bus' or 'Public Transport' ended up being hugely important during model training. This essentially answered my main academic question: yes, machine learning can pull behavioral insights out of boring, stagnant demographic data if you know how to engineer it.4.2 Sample Code WalkthroughHere are a few selected code snippets to show the core Software Engineering logic I implemented. These samples deal with the actual data cleaning and system security, rather than just simple Python scripts.Sample 4.2.1: Feature Engineering for Survival Bias (Data Preprocessing)A really critical step during the clean-up phase was removing "Survival Bias" from the raw employee dataset. If you give a machine learning algorithm an employee's *current chronological age*, you are letting the algorithm cheat. It will falsely learn that older employees have a higher probability of staying, simply because they survived in the company long enough to get old. To fix this, I engineered a brand-new feature (`Age_At_Hiring`) using Pandas, and then forcefully dropped the biased column.Sample 4.2.2: Hyperparameter Tuning Protocol (Machine Learning Tier)To ensure the Random Forest algorithm didn't just blindly memorize the training data (a major problem known as Overfitting), I built a `RandomizedSearchCV` cross-validation loop. This script randomly tests different combinations of tree depths and estimator counts. Crucially, I forced the model to evaluate itself using the `scoring='recall'` argument. As I talked about in the Academic Findings (Section 4.1), this one line of code forces the algorithm to mathematically care more about catching "True Positives" (leavers) instead of just padding its accuracy score.Sample 4.2.3: Secure Database Authentication Lifecycle (Presentation Tier / UI)Making a globally viable HR artifact means you have to take data security seriously. For the Streamlit User Interface, I blocked unauthorized access using a credentialing system hooked up to `mysql.connector`. Passwords are never saved as plain text. Instead, I used the `hashlib` library to scramble them cryptographically using `SHA-256` hashing before checking them against the MySQL database.4.3 Test CasesI ran a bunch of comprehensive functional tests to make sure the application architecture was stable before I called the deployment a success.4.3.1 Login (Test ID: 01)Test Case Identifier01Test Case  LoginTest Designed byRidmi WeerasingheTest Designed Date24th January 2026DescriptionUser enters the HR credentials (Email and Password). By establishing a formalized "Early Warning System" specifically targeted at predicting Infant Attrition (0-12 months) in the Sri Lankan garment industry, the project directly answered the foundational academic question. It proved that historical Demographic and Graphical profiles even in the absence of explicit financial or behavioural data house deeply embedded warning signals regarding employee retention. Furthermore, the project successfully mitigated the pervasive "Accuracy Paradox" inherent to HR turnover analytics. By explicitly tuning the Random Forest classifier to prioritize 'Recall' (Sensitivity) across an imbalanced array, the organizational value of the tool scaled dramatically, empowering HR professionals to accurately catch true-positive flight risks rather than resting on inflated, deceptive accuracy scores.5.2 LimitationWhile the artifact operates robustly within its targeted parameters, several distinct software and data limitations map to the boundaries of the achieved scope.Scope Achieved:The project fully achieved the integration of a three-tier architecture (Database, Predictive Back-End, Streamlit Presentation Front-End) operating offline entirely via historical demographic uploads. It definitively proved that geographic metrics (Transport profiles, specific localized Districts) operate successfully as proxy models for assessing economic stress-driven turnover.Scope Not Achieved (Limitations):Behavioural Tracking: Due to strict, justifiable organizational privacy and security regulations enforced by the data sponsor, live transactional data (such as monthly efficiency percentages, precise salary fluctuations, or absenteeism logs) could not be retrieved. Consequently, the predictive model operates strictly as a static demographic profiling tool, unable to detect sudden behavioural shifts indicative of "rage quitting" or instantaneous dissatisfaction.Dynamic Database Writing: While the MySQL database currently records metrics regarding Bulk Uploads, it does not currently store and update individual employee profiles over time within a relational table array, requiring HR to rely heavily on uploading localized CSV spreadsheets for single-batch runs.Hardware Localisation: Because the model relies on Scikit-Learn libraries and Streamlit operating locally, it limits instantaneous cross-branch access compared to a fully cloud-hosted Amazon Web Services (AWS) or Azure deployment.5.3 Critical EvaluationCritically evaluating the findings of this project highlights specific insights highly relevant to the broader HR-Analytics research community, particularly for practitioners operating in developing socio-economic environments.The project reinforces the premise that generic, open-source western datasets (e.g. standard IBM HR Attrition sets) often establish a false dichotomy of retention metrics. Many global researchers incorrectly assert that raw compensation and explicit "Job Satisfaction Surveys" are universally mandatory to generate predictive models. This project's findings sharply contest that narrative. By applying data-driven techniques specifically focused on localized metrics like 'Transport Mode', the model proved that environmental and logistical difficulty acts as a massive overriding variable in geographically dispersed, labour-intensive environments. The research community must therefore aggressively pivot toward hyper-localized data modelling incorporating geographically relevant constraints rather than relying on generalised retention matrices. Additionally, the project evaluated that implementing complex algorithms (like Stacking Ensembles or Neural Networks) yields diminishing returns in practical organisational HR deployments where explainability via interactive interfaces is paramount to user trust.5.4 Future WorkThe current artifact serves as an academic Proof of Concept (PoC), establishing a mathematically sound baseline for an HR predictive system within a localized development environment.
